# フロントエンド MOCK 方案の各モジュールにおける難易度・リスク比較

## 要点

- **サービスと ERP の連携規模の比較**  
  - 012C3 / 01206 / 01288 のインターフェース数、データ取得元、Pipe タイプ、マスタデータの退路などを表で比較。
- **Sms01206 / Sms01288 が MOCK とパラメータで回避しやすい理由**  
  - **01288**：judgmentERP と save 内の R3 移行のみが ERP に依存。マスタデータに FLAG "ERP" を設定していない。setFactoryResults / getUserList なし。回避ポイントが少ない。  
  - **01206**：通常モードでは 5 つの取得処理はまず SMS を利用。KEY1/KEY2="ERP" を渡さない場合は ERP に接続しない。setFactoryResults、moveR3、judgmentERP のみ回避または MOCK が必要。
- **Sms012C3 が実施難易度・リスクともに高い理由**  
  - 13 インターフェースすべてで MOCK が必要で強結合。呼び出しチェーンが多く、全業務ロジックをカバーする MOCK データの設計が困難。実施範囲が広く難易度が高い。
- **フロントエンド vs サービス側 MOCK データ量の見積もり（Sms012C3、全シナリオカバー）**  
  - **フロントエンド API 側**：1 本のチェーンを通す最小構成で **10～12 グループ**。中規模（例：得意先 5×機種 10）で全分岐をカバーする場合は **400～500 グループ**、規模拡大で **1500 グループ以上**（取得系 10 インターフェースのみ）。  
  - **サービス側 MOCK ERP（Pipe の反対側）**：最小で **10 グループ以内**（7 種コマンド各 1 グループ）。同程度の中規模で全シナリオをカバーする場合は **350～700 グループ**。71/72 の 1 セットで複数インターフェースを共有でき、保守ポイントが集約される。

## 一、3 サービスと ERP の連携規模比較

| 観点 | Sms012C3 | Sms01206 | Sms01288 |
|------|----------|----------|----------|
| **ERP 依存が明確なインターフェース数** | **13 個** | 通常モードでは **3 種**（送信 2＋判定 1）。取得 5 は「まず SMS、条件で ERP」 | **2 種**（judgmentERP ＋ save 内 R3 移行） |
| **取得系（マスタデータ）** | 10 個、いずれも ERP | 5 個、**通常モードでは SMS のみ**（画面で KEY1/KEY2="ERP" を渡さない） | 明確な取得系 0（マスタは SMS タイプ） |
| **送信系** | moveR3 ＋ **setFactoryResults** | moveR3 ＋ setFactoryResults | save 内の moveR3 のみ（setFactoryResults なし） |
| **判定系** | judgmentERP | judgmentERP | judgmentERP |
| **使用 Pipe タイプ** | 71 / 72 / 74 / 87 / 63 / 62・64 | 71 / 74 / 87 / 63 / 62・64（取得は試作出荷モード時のみ Pipe 経由の可能性） | 62・64 のみ（明示） |
| **得意先・配送先・担当の取得元** | **すべて ERP（72）** | SMS（CUSTOMER_LIST_1 / CONSIGN） | SMS（CUSTOMER_LIST_1 / CONSIGN） |
| **getUserList（最終ユーザ）の有無** | **あり（71+72）** | なし | なし |
| **getCustomerProduct（単品目）の有無** | **あり（87）** | なし（内部にはあるが 5 取得系では非公開） | ありだがタイプは CUSTOMER_PRODUCT（非 ERP） |
| **マスタ getMaster の明示的 FLAG "ERP"** | **あり、複数箇所** | **呼び出し側が KEY1/KEY2/KEY3="ERP" を渡す場合のみ**（通常画面では渡さない） | **なし**（PRICE / PRODUCT_LIST_1 等） |

Sms012C3 の ERP 接触面は他 2 つよりはるかに大きい：**インターフェース数が多く、Pipe 種類も多く、重要マスタ（得意先/配送先/担当）に「SMS マスタで代替」する退路がない**。このため「少数パラメータ＋局所 MOCK」だけでは ERP を体系的に避けにくい。**Sms01206** は通常受注モードでは 5 取得系は ERP に接続しない（ソース結論は《Sms01206_ERP_API_依赖关系图》第四節参照）ため、01288 と同様に ERP なしでも業務を維持しやすい。

## 二、Sms01206 / Sms01288 が MOCK とパラメータで回避しやすい理由

### 2.1 Sms01288：回避ポイントが少なく、マスタを ERP 経由にしなくてもよい

- **明確に ERP とやり取りが必要なのは 2 種のみ**：  
  - **judgmentERP**（ERP 連携機種かどうか）；  
  - **save / saveNewPackage 内で発火する moveR3 → moveOrderToR3**（R3 移行）。
- **マスタ**（getUnitPrice、getProducts、getCustomers、getConsigns 等）の実装では **PRICE、PRODUCT_LIST_1、CUSTOMER_LIST_1、CUSTOMER_CONSIGN** 等を使用しており、**FLAG "ERP" を設定していない**。このため **SMS マスタやキャッシュ**から取得する余地があり、必ずしも ERP に接続しない。
- **回避戦略**は単純：  
  - **judgmentERP** を MOCK で「非連携」または固定値を返すようにし、画面が「必ず R3 移行する」分岐に入らないようにする；  
  - または「R3 移行を実行しない」保存パスで操作する（業務上許容される場合）、**moveOrderToR3 は発火しない**；  
  - マスタが共通層で ERP 経由でなければ、**マスタ用の MOCK は不要**。SMS  DB と整合していればよい。
- **setFactoryResults なし**：「試作出荷実績登録」ボタン/フローが存在せず、MOCK または無効化が必要な送信ポイントが 1 つ少ない。
- **getUserList / getCustomerProduct（ERP 版）なし**：Pipe 71+72 に依存する最終ユーザ一覧も、明示的な ERP 単品目検索もないため、**整合を保つ MOCK の次元が少ない**。

したがって：**「judgmentERP の MOCK ＋ R3 移行実行の有無の制御」と「マスタがそもそも ERP 経由でない可能性」**により、Sms01288 は ERP が全滅しても業務を概ね実行しやすく、**難易度は低く、リスクは相対的にコントロール可能**。

### 2.2 Sms01206：通常モードでは 5 取得系は ERP に接続せず、送信・判定のみ回避

- **ソース結論**（《Sms01206_ERP_API_依赖关系图》第四節参照）：  
  - **getPrice / getUnitPrice / isUnitPriceMaster / getProducts** はいずれも **まず getMaster(PRICE または PRODUCT) で FLAG なし**。呼び出し側が **KEY2="ERP" または KEY1="ERP"** を渡し、かつ SMS にデータがない場合にのみ、FLAG "ERP" を付けて ERP にアクセスする。  
  - 画面 **OrderEntryNo出荷依頼表改造.java** では **試作出荷モード**（`isFactoryNumberUpdate()`）のときのみ上記インターフェースに KEY1/KEY2="ERP" を渡す。**通常受注画面では渡さない**ため、この 4 取得系は ERP なしでも **SMS のみで利用可能**。  
  - **getCustomerProducts** は **SMSMasterServer.CUSTOMER_PRODUCT** のみ使用し、FLAG "ERP" や ERP 分岐は一切ない。**常に SMS のみ**。  
  - **getConsigns / getCustomers** は **CUSTOMER_CONSIGN、CUSTOMER_LIST_1** を使用し、**SMS マスタ**由来。
- **したがって通常モードでは**、実際に ERP に接続し、回避または MOCK が必須なのは **3 種**のみ：**setFactoryResults**、**moveR3**、**judgmentERP**。
- **回避戦略**：  
  - **マスタ**：通常モードでは **5 取得系に対する MOCK は不要**。すでに SMS 経由。**試作出荷モード**で SMS にデータがない場合にのみ ERP にいく。その場合は ERP なしでは補完検索できないため、そのモードに入らないか MOCK を検討。  
  - **判定**：**judgmentERP** を MOCK するか、「ERP 連携機種かどうか判定不可」を受け入れる。  
  - **送信**：**パラメータまたは画面制約**（「試作出荷実績」「R3 移行」を押さない等）、または **setFactoryResults / moveR3** を MOCK で「成功」とし「実際には ERP に送っていない」ことを明示すれば、SMS 内でフローを通せる。
- **getUserList なし、getCustomerProduct（公開単品 ERP）なし**：012C3 より ERP 強依存が 2 つ少なく、MOCK の次元が少ない。

したがって：**通常受注モードでは 5 取得系の MOCK は不要**。**judgmentERP ＋ 送信 2 種**のみ回避または MOCK すれば、Sms01206 は ERP 不可時でも業務を概ね実行でき、**難易度・リスクとも低い**（01288 に近い）。

## 三、Sms012C3 が実施難しくリスクも高い理由

- **13 インターフェース、取得系 10 個がいずれも ERP 経由**。得意先/配送先/担当に SMS の退路がなく、**getUserList**（最終ユーザ）、**getCustomerProduct**（単品目）、**setFactoryResults**（試作出荷実績）が独自にあり、回避ポイントが分散し次元も多い。
- **API 呼び出しチェーンの結合度が高く**、チェーン数は業務ステップに応じて指数関数的に増える。各チェーンごとに制約の厳しい MOCK データを設計しないと、どこか 1 つでも不整合で保存失敗・移行失敗・後続ロジック誤りになる。**実施範囲が広く、難易度・リスクともに高い**。

## 四、フロントエンド API 呼び出し側で MOCK する際の難点（Sms012C3 を例に）

以下は **フロントエンド/API 呼び出し側**で Sms012C3 のインターフェースを MOCK する（getCustomers、getConsigns、getUserList、getPrice、moveR3 等 13 個の RMI インターフェースをゲートウェイまたはフロント層でインターセプトし、手書きデータを返す）場合の典型的な難点である。

**シナリオ一：受注入力から保存/R3 移行まで——複数インターフェースの直列、1 箇所の不整合でチェーン断絶**

- ユーザ操作の流れは一般的に：担当選択 → **getCustomers** → 得意先選択 → **getConsigns** → 機種選択 → **getProducts** → 販価 **getPrice/getUnitPrice** → **judgmentERP** → 保存 → **moveR3**。フロントは上記複数 API を**順次**呼ぶ。
- **フロント API 呼び出し側**でインターフェースごとに MOCK する場合：  
  - **getCustomers** が返す得意先コードは、**getConsigns** の MOCK で引数として検索できる配送先と対応している必要がある；  
  - **getConsigns** の配送先、**getProducts** の機種は、**getPrice/getUnitPrice** の MOCK の (得意先, 機種) 次元と一致している必要があり、そうでないと販価が带出されないかエラー；  
  - **moveR3** の MOCK は、前述ステップで組み立てた注文構造（法人/得意先/機種/販価等）を受け入れる必要がある。getCustomers、getProducts、getPrice 等の返すコードや形式と不整合だと、保存/移行でエラーまたはロジック異常になる。
- **難点**：フロント側で維持するのは**呼び出しチェーン全体**の複数 MOCK データの一貫性。1 つのインターフェースの戻り値（例：得意先を 1 つ増やす）を変えると、getConsigns、getPrice、moveR3 等の MOCK を同時に修正する必要があり、どこか 1 つでも不整合だと画面エラーや保存失敗になる。**多インターフェース強結合・フロント側 MOCK コストが高い**典型である。

**シナリオ二：同一インターフェースで複数組の MOCK データが必要**

- 同一インターフェースでも、引数や業務分岐によって画面ロジックが変わる。「単点」MOCK（例：1 組の固定データを返す）だけでは 1 パスしかカバーできず、他パスはエラーや異常表示になる。このため**同一インターフェースでも複数組の MOCK データ**が必要で、引数やシナリオに応じて切り替えて返す。
- **例一：getPrice / getUnitPrice**  
  引数は (法人, 得意先, 機種)（または 機種/得意先 の順は実装による）。画面は「販価の有無」「販価がゼロか」で分岐：有効な販価があれば価格带出、販価ゼロの場合は「無償出荷以外販価 ZERO」等のメッセージ、販価なしの場合は「販価マスタ未登録」。この 3 パターンをカバーするには少なくとも 3 組の MOCK：  
  - (法人A, 得意先X, 機種P) には有効な販価を返す；  
  - (法人A, 得意先Y, 機種Q) にはゼロまたは無償を示す販価区分；  
  - (法人A, 得意先Z, 機種R) には空/データなし。  
  「担当ごとに得意先一覧が異なる」による複数の (得意先, 機種) 組合せまでカバーすると、MOCK 組数はさらに増える。
- **例二：judgmentERP**  
  引数は機種コード。戻り値は ERP 連携機種かどうか。true のときは後続で R3 移行（moveR3）が走り、false のときは移行しない。「移行する」と「移行しない」の両方の操作パス（例：移行ボタンの有効/無効、保存後に moveR3 が呼ばれるか）を検証するには、少なくとも 2 組の MOCK：機種 M1 で true、機種 M2 で false。さらに前段の getProducts/getPrice 等の MOCK でも M1/M2 を用意し、チェーン全体が通る必要がある。
- **難点**：同一インターフェースの複数組 MOCK は「引数 → 戻り値」の対応だけでなく、他インターフェースの MOCK とデータ次元で一致している必要がある（例：judgmentERP で使う M1/M2 は getProducts の MOCK に存在すること）。そうでないとチェーンが切れるか、目的分岐に到達しない。

## 五、取得系 10 インターフェースの MOCK 組数見積もり（全シナリオカバー）

以下は Sms012C3 の**業務・操作ロジック**（コードと依赖関係図）に基づき、取得系 10 インターフェースを**フロント API 側で MOCK** する際に必要な「MOCK 組」数を見積もる。**1 組**とは：ある一定の引数（または引数組合せ）の下で、1 つの業務分岐または 1 つの選択パスをカバーするために用意する 1 種類の戻り値。

### 5.1 各インターフェースの引数次元と分岐

| インターフェース | 主な引数次元 | 戻り値の形態 | 「全シナリオ」カバーに必要な MOCK 組の区分数 |
|------------------|---------------------------|--------------|----------------------------------------------|
| **getChargeCustomers** | 法人、担当コード | リスト（得意先・名称・担当） | **担当**で区切り：担当 1 つあたり 1 組。担当数 = C_charge。 |
| **getCustomers** | 法人、検索キー(key)、KEY1(機種) | リスト（得意先・名称・担当） | 2 パス：① 担当/検索キー LIKE（Z001）→ key 形態 1 種あたり 1 組；② 機種 getCustomers2（CUSTOMER_PRODUCT_ERP2）→ 機種 1 つあたり 1 組。key 形態数 = K_cust、機種数 = P。 |
| **getConsigns** | 法人、得意先 | リスト（配送先等） | **得意先**で区切り：得意先 1 つあたり 1 組。得意先数 = N_c。 |
| **getProducts** | 法人、機種検索キー、KEY1(得意先[]) | リスト（機種・名称・品種等） | 2 パス：① 検索キー（%/前方/完全一致）→ 検索キー 1 種あたり 1 組；② 代理店用に得意先で getProductsDairiten → 得意先 1 つあたり 1 組。検索キー形態数 = K_prod、代理店検索に使う得意先数 = N_d。 |
| **getCustomerProducts** | 法人、KEY1(得意先[]) | リスト（機種・名称） | 内部は getProductsDairiten。**得意先**で区切り：得意先 1 つあたり 1 組。同上 N_d。 |
| **getPrice** | 法人、機種、得意先 | 1 件（14+1 列：販価・通貨・相手機種名等） | **(法人, 得意先, 機種)** で区切り；**データあり/なし**等の分岐も必要。有効 (得意先, 機種) 組合せ数 = N_c × P、さらに分岐数 B_price（有/無 ≈ 2）を乗じる。 |
| **getUnitPrice** | 法人、機種、得意先、有効日 | 1 件（販価・通貨 2 列） | 同上 key；**新販価/現販価**（s[8] と yyyymmdd 比較）、**データなし**もカバー。分岐数 B_unit ≈ 3。 |
| **isUnitPriceMaster** | 法人、機種、得意先 | 真偽 | 同上 key；**true / false** の 2 分岐。key あたり 2 組、key 数 = N_c × P。 |
| **getCustomerProduct** | 法人、得意先、機種（prm の KEY） | 1 件（品目明細） | **(法人, 得意先, 機種)** で区切り；データあり/なし。約 N_c × P × 2。 |
| **getUserList** | 法人(key[0])、現在の最終ユーザ(KEY1) | リスト＋現在行（最終ユーザリスト） | **法人**で区切ればよい（内部で 71+72 でリストを組み立て）。法人數 = Corp。 |

注：上記「得意先数 N_c」「機種数 P」等は、**MOCK で区別し、チェーンに参加させる**マスタ件数であり、実システムの総数ではない。かつ各インターフェース間で**同一の**法人/担当/得意先/機種コードを共有する必要があり、そうでないとチェーンが切れる。

### 5.2 1 本のチェーンを通す最小構成（単一路径）

- 法人 1、担当 1、得意先 1、配送先 1、機種 1、販価 1 組（データあり）、getUserList 1 組、getCustomerProduct は任意で 1 組。  
- **概算：** getChargeCustomers 1 ＋ getCustomers 1～2 ＋ getConsigns 1 ＋ getProducts 1～2 ＋ getCustomerProducts 1 ＋ getPrice 1 ＋ getUnitPrice 1 ＋ isUnitPriceMaster 1 ＋ getCustomerProduct 0～1 ＋ getUserList 1  
- **合計約 10～12 組**（「担当選択 → 得意先選択 → 配送先選択 → 機種選択 → 販価带出 → 保存」の 1 パスのみを通すことを保証）。

### 5.3 「分岐」と「複数選択」をカバーした後の見積もり式

**チェーン上のデータ一貫性**を保ったうえで（同一の担当/得意先/機種を各インターフェースで再利用）、必要となるのは：

- **getChargeCustomers**：C_charge 組（担当あたり 1 組のリスト）。
- **getCustomers**：K_cust 組（担当/検索キー）＋ P 組（機種 getCustomers2）、計 **K_cust + P** 組。
- **getConsigns**：**N_c** 組（得意先あたり 1 組の配送先）。
- **getProducts**：K_prod 組（検索キー）＋ N_d 組（代理店・得意先）、計 **K_prod + N_d** 組。
- **getCustomerProducts**：**N_d** 組（得意先あたり 1 組の品目リスト）。
- **getPrice**：**(N_c × P) × B_price** 組（B_price は ≥ 2 推奨：有/無）。
- **getUnitPrice**：**(N_c × P) × B_unit** 組（B_unit は ≥ 3 推奨：新販価/現販価/無）。
- **isUnitPriceMaster**：**(N_c × P) × 2** 組（true/false）。
- **getCustomerProduct**：**(N_c × P) × 2** 組（有/無、NSCM 品目）。
- **getUserList**：**Corp** 組（法人あたり 1 組の最終ユーザリスト）。

**合計（オーダー）：**  
2・(N_c × P) + (N_c × P)・(B_price + B_unit) + (N_c × P)・2 + (N_c × P)・2 + (K_cust + P) + (K_prod + N_d) + N_c + N_d + C_charge + Corp  
≈ **(N_c × P)・(4 + B_price + B_unit) + K_cust + K_prod + P + 2・N_d + N_c + C_charge + Corp**。

**中規模**の例で代入：N_c = 5、P = 10、C_charge = 3、Corp = 2、K_cust = 2、K_prod = 3、N_d = 3、B_price = 2、B_unit = 3：

- (N_c×P)・(4+2+3) = 50×9 = **450**
- K_cust + K_prod + P + 2・N_d + N_c + C_charge + Corp = 2+3+10+6+5+3+2 = **31**
- **合計約 480 組**（取得系 10 インターフェースのみ。judgmentERP / moveR3 / setFactoryResults の MOCK は含まない）。

### 5.4 サービス側 MOCK ERP（Pipe の反対側）で全シナリオをカバーするのに必要なデータ組数

**サービス側**で MOCK する場合、フロント/R3Server は「getPrice / getCustomers」等 13 インターフェースを直接呼ばず、**R3Server / sms012C3_s** が **Pipe コマンド**（71 / 72 / 74 / 87 / 62 / 64 / 63）で MOCK サービスと通信する。このため MOCK 側は**コマンド番号＋リクエストパラメータ**に応じて規定フォーマットのメッセージを返せばよく、**1 組のデータ**は：あるコマンドがあるパラメータクラスに対する 1 つの戻り値（1 行または複数行）に対応する。

- **71（販価）**：リクエストは (corp, customer, product, date 等)。getPrice / getUnitPrice / isUnitPriceMaster はいずれも 71 経由で、**同一の 71 データを共有**でき、3 インターフェースごとに別々に用意する必要はない。全シナリオを区別する組数 = **N_corp × N_c × P** × 応答タイプ数（有/無/新販価・現販価 等、約 2～3）。  
  例：2×5×10×2 = **200 組**。
- **72（得意先）**：リクエストは (corp, customer, opt, flg)。flg に Z001（配送先等）、Z002、Z007（最終ユーザ）がある。getConsigns / getCustomers / getEndUser はいずれも 72 経由で、**同一の 72 データを共有**。(corp, customer, opt, flg) で区切ればよい。組数 ≈ **N_corp × N_c × (opt 数 × flg 数)**。opt は EQ/LIKE、flg は Z001/Z002/Z007 が一般的で約 2×3 = 6。  
  例：2×5×6 = **60 組**。
- **74（機種）**：リクエストは (corp, plant, product, opt)。getProducts / judgmentERP が 74 を間接利用。**74 の 1 セット**で足りる。組数 ≈ **N_corp × N_plant × P × 2**（EQ/LIKE）。N_plant は多くの場合 1。  
  例：2×1×10×2 = **40 組**。
- **87（得意先品目）**：リクエストは (corp, customer, cust_opt, product, prod_opt)。getCustomerProducts / getCustomerProduct はいずれも 87 経由で、**87 データを共有**。組数 ≈ **N_corp × N_c × P × (cust_opt × prod_opt)**、約 2×2 = 4。  
  例：2×5×10×4 = **400 組**（opt で区切らない場合は 100 組程度に圧縮可能）。
- **62 / 64 / 63**：送信系。「成功/失敗」等の分岐でよい。各 2 組程度、計約 **6 組**。

**合計（5.3 と同規模 N_corp=2, N_c=5, P=10, N_plant=1）：**  
71 約 200 ＋ 72 約 60 ＋ 74 約 40 ＋ 87 約 100～400 ＋ 62/64/63 約 6 → **約 406～706 組**。87 を opt で細かく分けず、71 を有/無の 2 種応答のみにすると **約 350～450 組**程度になる。

**フロント API 側**との比較：同一業務規模では、フロント取得系 10 インターフェースで約 **480 組**（5.3 参照）、サービス側で約 **350～700 組**。サービス側では getPrice / getUnitPrice / isUnitPriceMaster の 3 インターフェースのために 71 データを 3 副本作る必要はなく、getConsigns / getCustomers / getEndUser のために 72 を複数副本する必要もないため、**組数はやや少ないか同程度**。ただしサービス側では 71/72/74/87 の**メッセージフォーマット**（列数、区切り、順序）が R3Server の解析と一致している必要があり、そうでないと NPE やずれが発生する。

### 5.5 結論

- **1 本のチェーンを通す最小**：フロント取得系 10 インターフェースで約 **10～12 組**；サービス側 Pipe は **7 種コマンド × 各 1 組 ≈ 10 組以内**（単一法人/単一得意先/単一機種/単一路径）。
- **全シナリオカバー（中規模）**：  
  - **フロント API 側**：取得系 10 インターフェースで約 **400～500 組**。N_c×P を大きくすると 1500 組超もあり得る。  
  - **サービス側 MOCK ERP**：71/72/74/87/62/64/63 合計で約 **350～700 組**（N_c×P に依存し、87 を opt で区切るかで上限が変わる）。  
- サービス側 MOCK の組数は**フロントと同オーダー**だが、**71 の 1 セットで getPrice / getUnitPrice / isUnitPriceMaster を同時に満たせ**、**72 の 1 セットで getConsigns / getCustomers / getUserList 内の getEndUser を同時に満たせる**ため、保守ポイントが集約される。「全シナリオをカバー」するには、いずれも**数百組**のデータが必要で、judgmentERP、moveR3 等のロジックとデータ次元で一致している必要がある。
